Git: Controle de Versão

Nesta aula, vamos aprender sobre o Git, um sistema de controle de versão essencial para desenvolvedores web modernos. O Git permite que você acompanhe as mudanças no código, trabalhe em equipe de forma eficiente e mantenha um histórico completo do seu projeto.

História e Conceitos Básicos do Git

Origem e Criação

  • Criado por Linus Torvalds em 2005 para o desenvolvimento do kernel Linux
  • Sistema de controle de versão distribuído (DVCS)
  • Alternativa a sistemas centralizados como SVN e CVS

Conceitos Fundamentais

Repositório
Armazenamento de arquivos e histórico de mudanças.
Commit
Snapshot do projeto em um momento específico.
Branch
Linha independente de desenvolvimento.
Merge
Combinar alterações de diferentes branches.
Clone
Cópia completa de um repositório.
Pull/Push
Sincronização de mudanças entre repositórios.

Fluxo Visual do Processo Git

A imagem acima ilustra um fluxo simplificado do versionamento com Git:

  1. Daniel cria a versão inicial (v1) do documento e envia para o servidor remoto via git push
  2. Matheus baixa a versão v1 do documento usando git pull
  3. Matheus modifica o documento, criando a versão v2, e envia para o servidor via git push
  4. Daniel atualiza seu repositório local com a versão v2 usando git pull
Importante

O servidor Git (GitHub, GitLab ou Bitbucket) mantém o histórico completo de todas as versões, permitindo rastrear a evolução do projeto e identificar quem fez cada alteração.

Processo de Rollback

Para reverter mudanças problemáticas:

  1. Uma versão v3 é criada e enviada ao servidor
  2. A equipe detecta falhas nesta versão
  3. É executado um git reset para retornar à versão v2 estável
  4. A versão corrigida pode então ser desenvolvida a partir da v2
Comando de Rollback
git reset --hard a1b2c3d

Git Flow Simplificado: Organização de Branches

A imagem acima ilustra o Git Flow simplificado, um modelo de fluxo de trabalho estruturado para projetos Git. Este modelo estabelece uma organização clara de branches e processos que facilitam o desenvolvimento colaborativo e mantêm a integridade do código.

Componentes do Git Flow

Os principais componentes deste fluxo são:

Branch Main
É a branch principal que contém o código de produção estável. Representa o que está atualmente em produção.
Branch Develop
É a branch de desenvolvimento onde novas funcionalidades são integradas antes de irem para produção.
Feature Branches
São branches criadas a partir da develop para implementar novas funcionalidades.
Hotfix Branches
São branches criadas para correções rápidas ou urgentes que precisam ir direto para produção.

Sincronização e Regras do Git Flow

É fundamental que as branches main e develop sejam sempre mantidas sincronizadas – com exceção dos períodos em que novas funcionalidades estão sendo desenvolvidas na develop e ainda não integradas à main.

Regras Importantes
  1. As feature branches são sempre criadas a partir da branch develop, nunca da main.
  2. Nunca fazemos merge direto das feature branches na main; o merge ocorre primeiro na develop.
  3. Os merges são realizados via pull request, que idealmente é revisado por outro desenvolvedor.
  4. As hotfix branches são criadas a partir da main para correções urgentes que não podem seguir o fluxo padrão.
  5. Após o merge de uma hotfix na main, é necessário mergear essa hotfix na branch develop para manter a sincronização.
  6. Todos os merges são feitos através de pull requests para garantir revisão de código e qualidade.
Benefícios

Esse fluxo organizado ajuda a reduzir conflitos, facilita a identificação de problemas e permite que equipes trabalhem simultaneamente em diferentes funcionalidades sem interferir umas nas outras. Além disso, garante que o código em produção seja sempre estável.

Referência: Para um estudo mais aprofundado sobre o GitFlow, consulte o tutorial oficial da Atlassian: https://www.atlassian.com/git/tutorials/comparing-workflows/gitflow-workflow

Começando com GitHub

Criar conta no GitHub

Crie sua conta no GitHub para hospedar seus repositórios e colaborar com outros desenvolvedores.

  1. Acesse github.com
  2. Clique em "Sign up" e siga o processo de criação de conta
  3. Confirme seu e-mail

Adicionar chave SSH ou Configurar HTTPS

Conexão via SSH

Chaves SSH oferecem uma conexão segura sem a necessidade de inserir a senha a cada operação.

Gerar nova chave SSH
ssh-keygen -t rsa -C "seu.email@exemplo.com"
Copiar chave (Linux/macOS)
cat ~/.ssh/id_rsa.pub
Copiar chave (Windows - Git Bash)
cat ~/.ssh/id_rsa.pub

Adicione a chave pública à sua conta em Settings > SSH and GPG Keys.

Conexão via HTTPS

A conexão HTTPS é simples de configurar, mas requer autenticação a cada push.

Clonar repositório via HTTPS
git clone https://github.com/usuario/repositorio.git

Você também pode utilizar o GitHub Desktop para uma interface gráfica.

  1. Baixe e instale o GitHub Desktop em desktop.github.com
  2. Faça login com sua conta GitHub
  3. Utilize a interface para gerenciar seus repositórios

Criar repositório no GitHub

Após criar sua conta, crie um novo repositório:

  1. Clique no botão "+" e selecione "New repository"
  2. Dê um nome ao repositório
  3. Adicione uma descrição (opcional)
  4. Escolha se será público ou privado
  5. Selecione "Initialize this repository with um README" (se desejar)
  6. Clique em "Create repository"

Comandos e Operações do Git

Configuração Inicial

Configure seu nome de usuário e e-mail:

Nome de usuário
git config --global user.name "Seu Nome"
Email
git config --global user.email "seu.email@exemplo.com"
Verificar configurações
git config --list

Iniciando um Repositório

Criar repositório
git init
Verificar status
git status

Adicionar e Commitar Arquivos

Adicionar arquivo específico
git add arquivo.txt
Adicionar todos os arquivos
git add .
Realizar commit
git commit -m "Mensagem descritiva do commit"
Verificar histórico
git log

Trabalhando com Branches

Listar branches
git branch
Criar novo branch
git branch nome-do-branch
Mudar para outro branch
git checkout nome-do-branch
Criar e mudar para branch (atalho)
git checkout -b nome-do-branch

Merge e Resolução de Conflitos

Merge
git merge nome-do-branch
Resolver conflitos e commit
git add arquivo-com-conflito
git commit -m "Resolução de conflitos"

Repositórios Remotos

Clonar repositório (HTTPS)
git clone https://github.com/usuario/repositorio.git
Clonar repositório (SSH)
git clone git@github.com:usuario/repositorio.git
Adicionar repositório remoto
git remote add origin https://github.com/usuario/repositorio.git
Push para repositório remoto
git push origin nome-do-branch
Pull do repositório remoto
git pull origin nome-do-branch
Listar remotos
git remote -v

Um fluxo de trabalho típico com Git segue estes passos:

Exercício Prático: TaskMaster

TaskMaster com Controle de Versão

Objetivo: Implementar controle de versão para o projeto TaskMaster.

Instruções:
  1. Inicialize um repositório Git na pasta do projeto TaskMaster: 
    git init
  2. Crie um arquivo .gitignore apropriado (use o exemplo acima).
  3. Adicione os arquivos iniciais (HTML, CSS, JS) e faça o primeiro commit: 
    git add .
git commit -m "Commit inicial: estrutura básica do TaskMaster"
  4. Crie um branch de desenvolvimento chamado dev: 
    git checkout -b dev
  5. No branch dev:
    • Adicione funcionalidade para criar novas tarefas
    • Faça commit das alterações: 
      git add .
git commit -m "Adiciona funcionalidade para criar novas tarefas"
  6. Crie um branch feature/delete-task a partir do dev: 
    git checkout -b feature/delete-task
  7. No branch feature/delete-task:
    • Implemente a funcionalidade para excluir tarefas
    • Faça commit das alterações: 
      git add .
git commit -m "Implementa funcionalidade para excluir tarefas"
  8. Volte para o branch dev e faça merge do branch feature/delete-task: 
    git checkout dev
git merge feature/delete-task
  9. Crie um repositório no GitHub e conecte-o ao seu projeto.
  10. Adicione o repositório remoto e envie os branches: 
    git remote add origin https://github.com/seu-usuario/taskmaster.git
git push -u origin main
git push -u origin dev
git push -u origin feature/delete-task

Exercícios para os Alunos

Exercício 1: Inicializar um repositório e criar o primeiro commit

Objetivo: Praticar a criação de um repositório Git e o primeiro commit.

  1. Crie uma pasta para o projeto TaskMaster
  2. Abra o terminal nesta pasta
  3. Inicialize um repositório Git
  4. Crie um arquivo README.md com uma breve descrição do projeto
  5. Adicione o arquivo ao staging
  6. Faça o commit com uma mensagem descritiva
Ver solução 

mkdir TaskMaster
cd TaskMaster
git init
echo "# TaskMaster" > README.md
echo "Um aplicativo de gerenciamento de tarefas desenvolvido para o curso de JavaScript." >> README.md
git add README.md
git commit -m "Adiciona arquivo README com descrição do projeto"

Exercício 2: Trabalhar com branches

Objetivo: Praticar a criação e navegação entre branches.

  1. Partindo do repositório criado no exercício 1
  2. Crie um arquivo index.html com uma estrutura básica
  3. Adicione e commit o arquivo
  4. Crie um novo branch chamado feature/estilo
  5. Mude para este branch
  6. Crie um arquivo style.css vazio
  7. Adicione e commit o arquivo
  8. Volte para o branch principal
Ver solução 

echo "" > index.html
echo "TaskMaster
TaskMaster
" >> index.html
git add index.html
git commit -m "Adiciona estrutura HTML básica"
git branch feature/estilo
git checkout feature/estilo
touch style.css
git add style.css
git commit -m "Adiciona arquivo CSS vazio"
git checkout main

Exercício 3: Merge de branches

Objetivo: Praticar a combinação de mudanças usando merge.

  1. No branch feature/estilo, adicione código CSS básico ao style.css
  2. Adicione e commit as alterações
  3. Volte para o branch principal
  4. Faça merge do branch feature/estilo para o principal
Ver solução 

git checkout feature/estilo
echo "body { font-family: Arial, sans-serif; margin: 0; padding: 20px; }" > style.css
echo "h1 { color: #333; }" >> style.css
git add style.css
git commit -m "Adiciona estilos básicos"
git checkout main
git merge feature/estilo

Exercício 4: Configurar um repositório remoto

Objetivo: Configurar e utilizar um repositório remoto.

  1. Crie um novo repositório no GitHub (sem inicializar com README)
  2. Conecte seu repositório local ao remoto
  3. Envie todos os commits para o repositório remoto
Ver solução 

# Após criar o repositório no GitHub:
git remote add origin https://github.com/seu-usuario/taskmaster.git
